### Scope Refinement

**Explicit In-Scope**:

  - Per-agent responsibility matrix (what each agent must produce, check, and sign off on at each phase).
  - Controller gating rules: when to stop, what to verify, how to delegate.

**Explicit Out-of-Scope**:
  - Changing the actual content of existing task reports.
  - Modifying any task code, implementation, or business logic.
  - Creating template files inside new task folders (structure-only, content generated by agents).
  - Enforcing the workflow retroactively on committed history.

**Boundary Conditions**:
  - New tasks: fully apply 10-phase workflow.
  - Existing tasks: the new convention applies to the next work done; existing files stay where they are but are referenced via a `README.md` index.
  - The workflow is advisory for user-initiated ad-hoc requests but **mandatory** for agent-initiated, multi-step work (anything with >1 agent delegation).
  - `AGENTS.md` updates are additive — existing sections preserved unless explicitly superseded.

**Edge Cases Identified**:
  - Task spanning multiple days: date prefix stays fixed as the task-origin date; daily changelogs use dated filenames within the task folder.
  - Failed phase: write a `phase_skipped.md` or `phase_blocked.md` with the reason, do not delete the phase folder.
  - Controller crash mid-phase: `checkpoint-resume` skill handles state recovery; the phase folder must contain sufficient artifacts for any agent to resume.
  - Agent produces wrong/no artifact: controller halts at next gateway_check and routes back to the responsible agent with a remediation note.
  - Parallel agents in same phase: use sub-artifact names (`01_slug.md`, `02_slug.md`) within the phase folder, merge via controller.
